예약 규칙 보강 명세서를 `docs/예약_규칙_명세_v1.md`에 작성함. 정본 [[carwash-reqdoc-conventions]](require_v1.md)을 대체하지 않는 **보강 문서**로 위치시킴.

**Why:** 개발자가 예약 플로우에 추가할 2개 규칙(순차 선택 게이팅, 차종↔베이 노출)을 구두 전달. 코드 수정 없이 기획 문서만 작성.

**How to apply:** 이 도메인의 후속 규칙 문서 작성 시 아래를 활용.
- 규칙 1(순차 게이팅): 매장→매니저→차종·베이→날짜/시간. 현행 `reserve.vue`(`v-if` 노출 + `canPickSlot`/`canReserve` computed)와 정합. 단 현행은 매장 선택 시 매니저·차종 동시 노출(엄격 직렬 아님) → 미해결 Q6.
- 규칙 2(차종↔베이): **현행 구현과 충돌**. 현행=베이 코드 매장별 일련번호 A1/A2/A3 + `Bay.size` 3등급(SMALL/MID/LARGE) + 누적 수용(`SIZE_RANK[b.size] >= min`, storeService.ts `getBaysForCar`). 신규=코드가 크기 의미(A=소/B=중/C=대/D=특대) + 4등급(특대 신설) + 1:1처럼 보임.
- 핵심 미해결 질문 8개(Q1~Q8): 5분류→4등급 매핑(Q1), 크기당 베이 1개vs복수(Q2), 1:1vs누적(Q3), A~D 코드와 매장별 데이터 마이그레이션(Q4), 특대형↔5분류 연결 및 가격표 영향(Q5), 규칙1 직렬 엄격성(Q6), 대행 적용(Q7), 빈 베이 목록 처리(Q8).
- 충실성 원칙 준수: 사용자 제공 사실(A1/B1/C1/D1, 소/중/대/특대)은 원문 보존, 불명확 매핑은 `?` 표기 + 미해결 질문으로. 임의 매핑 창작 금지.

**문서 구조(이 프로젝트 표준):** 출처 추적표 → 개요 → 목표/성공지표 → 대상사용자 → 요구사항(FR/NFR) → 상세명세(상태흐름도+매핑표) → 제약·가정 → 범위외 → 미해결질문. 표·상태흐름도 적극 사용은 [[carwash-reqdoc-conventions]]와 동일.
