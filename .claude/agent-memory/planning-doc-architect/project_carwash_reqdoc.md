자동차 세차 예약 서비스(MVP) 요구사항 정의서를 `docs/require_v1.md`에 작성함 (13개 섹션 구조).

**Why:** 개발자 원시 초안을 기획/이해관계자가 의사결정에 쓸 수 있는 정식 문서로 재편성하기 위함. FO 사용자 플로우 우선 구현, BO는 2차 문서화.

**How to apply:** 후속 버전(v2 등) 작성 시 아래 관례를 유지할 것.
- 프로세스 코드 표기법 `FW#/M#/S#`를 그대로 보존(초안 추적성). 교정 시 매핑 추적표를 함께 둠.
- 초안 오타 교정 확정: STMP→SMTP, FW4(매장예약)→FW5, F5→FW6(세차완료), F6→FW7(예약취소).
- 인증/승인 분리 원칙: "이메일 인증(전원 필수)" vs "승인(매니저 가입=관리자, 사용자 매장가입=매장 최고매니저)". 초안의 "승인 불필요" 문구와 모순되므로 이 분리로 해소함.
- FE 스택은 Nuxt 4(Vue 3 `<script setup>` + Vite 내장 + Nitro, Pinia, 파일 기반 라우팅)로 전환 완료·확정.
- 데이터 진화 3단계(프론트 더미 → Spring Boot 인메모리 → MySQL)에 동시성 처리를 매핑. 최종 방어선은 슬롯 UNIQUE(storeId,bayId,date,timeSlot).
- 표/상태머신 표 적극 사용이 이 프로젝트의 선호 형식.

**v1.3 추가 확정 규칙(2026-06-20):**
- 매니저 3교대 정의 신설(5.5): SHIFT_1 06~14, SHIFT_2 14~22, SHIFT_3 22~06(익일). 매장 운영 24시간은 변경 금지.
- 매니저 휴무 유형: FULL_DAY(그날 전체 슬롯 비활성) / 교대조 단위(SHIFT_1·2·3, 복수 가능, 해당 시간대 30분 슬롯만 비활성). '오전/오후 반차'를 3교대로 일반화.
- ManagerDayoff에 휴무유형 속성 추가(5.4). 부분 휴무도 동일 결재 흐름(8.2: 신청→최고매니저→관리자).
- 베이 슬롯 점유 선택불가(5.2·6.1·7장): 베이가 (날짜,시간) 슬롯에 RESERVED/COMPLETED면 그 슬롯에서 비활성. 선택순서 현행 유지(매장→매니저→차종→베이→날짜/시간), 날짜·시간 선택 후 점유 베이 비활성.

**v1.4 추가 확정 규칙(2026-06-20):**
- 예약 플로우를 3단계 화면 위저드로 분할(6.5 신설): 1p `/reserve`(매장→매니저→차종→서비스, 가격 자동표시) → 2p `/reserve/slot`(날짜·시간 → 베이 점유 그리드 선택, 예약하기 확정) → 3p `/reserve/done`(완료 요약).
- 진행 데이터는 Pinia 스토어 보관, 1단계 in-memory만(새로고침 시 초기화). 미완료 단계 직접 진입 시 1p로 되돌리는 가드. 2·3단계 서버/Redis/JWT 교체 경계(12.3에 1줄 연계).
- 베이 선택 버튼 그리드 단일화, 검색 select box(SearchableSelect) 베이 선택 폐지. 단 매장·매니저 검색 select+필터(6.3)는 유지.
- 선택 순서 갱신: v1.3의 "매장→매니저→차종→베이→날짜/시간"을 "매장→매니저→차종→서비스(1p) → 날짜·시간 → 베이 그리드(2p)"로 변경. 베이 점유 비활성 규칙(5.2/6.1) 자체는 유지, 순서 서술만 수정.
- 12.4 화면목록의 "예약 페이지" 1행을 3개 라우트 행으로 분할. 13.1 확정표에 #6/#7 추가.

**v1.6 추가 확정 규칙(2026-06-21) — 3역할 단순화 & 결재(승인) 전면 제외:**
- 역할 체계를 **사용자/매니저/관리자 3역할**로 단순화. **매장 최고권한 매니저(STORE_ADMIN) 역할 제거**. 이전 v1.5의 인증/승인 분리 원칙(line 15 참조)을 번복 — 이제 **승인 분기를 전면 제거**하고 모든 가입은 이메일 인증만으로 ACTIVE.
- 가입 상태머신 3상태로 축소: `REQUESTED → EMAIL_VERIFIED → ACTIVE` (PENDING_APPROVAL·REJECTED 제거).
- 매니저 범위 = 예약 대행(M3) + 세차완료 보조(M4)/취소 보조(M5)만. 관리자 = 매장별 관리·매출 확인(S4~S8). 권한 매트릭스 3열로 재작성.
- 휴일/휴무는 **2단계 승인 결재 제거**, 슬롯 가용성(§6.1)에만 반영되는 **도메인 데이터로만 유지**. StoreHoliday/ManagerDayoff의 '승인상태' 속성 제거(휴무유형은 유지). 등록 주체 추후 확정(§13.2로 강등).
- Manager 엔티티 `isStoreAdmin` 속성 제거.
- **보존 관례(중요):** 제거 대상 서술(§4.3 승인흐름표·§8.1~8.3 결재 워크플로우·상태머신·M6/M7/S3)은 삭제하지 않고 **(deprecated — v1.6) 주석으로 보존**. FW/M/S 번호도 보존(추적성). 이 프로젝트는 deprecated 보존 + 인용블록 메모 방식을 선호.
- §2.2 하단에 '⚠️ 구현-요구 정합성 이행 메모' 인용블록 추가: 2차 BE 구현(Phase 5~7)의 결재·STORE_ADMIN·M6/M7은 v1.6 요구 초과 → 후속 코드 정리 대상. M3+M4/M5는 부합 유지. (구현이 정의서를 앞서간 상황) [[project_roadmap_docs]]

**v1.7 추가 확정 규칙(2026-06-21) — 4역할 세분화 & 2단계 가입 승인 부활 (v1.6 번복):**
- v1.6의 3역할 단순화·승인 제거를 **전면 번복**. 역할 = `USER`(일반사용자)/`MANAGER`(일반매장매니저)/`STORE_ADMIN`(매장매니저관리자)/`ADMIN`(관리자) **4역할**. STORE_ADMIN 부활. 코드 현실(enum 4역할, `ApprovalStatus: SUBMITTED→APPROVED_L1→CONFIRMED→REJECTED`)과 재정합 성격.
- **단계 수 핵심 규칙(문서 전반 일관 유지 필수):** 가입 승인 = **2단계**(STORE_ADMIN 1차 M7 → ADMIN 2차 S3). 휴가/반차 승인 = **1단계**(STORE_ADMIN 종결 M8). 혼동 금지.
- 가입 상태머신 확장(§4.4): `REQUESTED → EMAIL_VERIFIED → PENDING_APPROVAL_L1 → PENDING_APPROVAL_L2 → ACTIVE` (+REJECTED). **ACTIVE만 로그인 가능** 명문화. USER는 EMAIL_VERIFIED→ACTIVE 직행.
- 권한 매트릭스(§3.2) 4열. STORE_ADMIN은 MANAGER 기능 상위호환 포함. 프로세스 코드 재정의: M6=휴가/반차 신청(구 매장예약승인 deprecated), M7=가입 1차 승인(부활), M8=휴가/반차 승인(신설), S3=가입 2차 최종승인(부활), S9=매니저 근무상태 확인(신설). ADMIN 범위는 매출(S8)·예약상태(S4)·후기(S6)·매니저 근무상태(S9)로 정리(구 S5 사용자관리·S7 회원정보확인은 매트릭스/코드표에서 비활성).
- §8 휴일/휴무: 휴가/반차 1단계 승인 결재 부활(§8.2~8.3). §8.3 상태머신 = `SUBMITTED → APPROVED / REJECTED`(1단계). 휴일 등록 주체는 여전히 미확정(§13.2).
- §12.4 화면목록 = **역할별 4그룹 분리표**. 경로는 "문서상 제안, 실제 라우트는 코드 시 확정"으로 명시. USER 그룹만 1차, BO 3역할은 2차.
- §13 신규 미해결: STORE_ADMIN 최초 1명 부트스트랩, ADMIN 가입 승인 정책, 휴일 등록 주체.
- **보존 관례 유지:** v1.6 서술은 삭제 않고 `~~취소선~~ *(deprecated — v1.6)*` 로 보존. §1.2 v1.6 행은 추적성 위해 그대로 둠. v1.6이 덮어쓴 서술 중 v1.7에서 되살아난 것은 deprecated 표기 풀고 v1.7로 갱신.

**미확정(v2에서 확정 대상):** 차종/서비스 분류 최종안, 가격표 실수치, SMTP 인프라, 알림 정책. 휴일 등록 주체, STORE_ADMIN 부트스트랩, ADMIN 가입 승인 정책. [[carwash-undecided]]
