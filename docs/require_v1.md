## 1. 문서 개요

### 1.1 문서 목적
개발자가 작성한 원시 요구사항 초안을 정식 요구사항 정의서로 재편성하여,
기획·개발·이해관계자가 동일한 기준으로 MVP 범위와 동작을 이해하고
의사결정에 활용할 수 있도록 한다.

### 1.3 용어 정의
| 용어 | 정의 |
|------|------|
| **FO** (Front-Office) | 사용자(고객)가 직접 사용하는 화면/플로우. 로그인·예약 페이지 등 |
| **BO** (Back-Office) | 매니저·관리자가 사용하는 관리 화면/플로우 (이번 MVP는 문서화만, 2차 구현). **(v1.6 범위 축소)** 매니저는 **사용자 예약 대행(+세차완료/취소 보조)**, 관리자는 **매장별 관리·매출 확인**으로 범위를 한정한다(결재/승인 워크플로우 제외) |
| **매장** (Store) | 세차 서비스를 제공하는 물리적 지점. N개의 베이를 보유 |
| **베이** (Bay) | 매장 내 개별 세차 라인. `A1 ~ AN`으로 식별. 동일 시간대 베이 수만큼 동시 예약 수용 가능 |
| **슬롯** (Slot) | 예약 가능 단위. `(매장, 베이, 날짜, 30분 시간단위)` = 1슬롯 |
| **결재 (= 승인)** *(v1.6 제외)* | 신청 건을 상위 권한자가 검토·확정하는 워크플로우. 본 문서에서 "결재"와 "승인"은 동의어로 사용. **v1.6에서 가입 승인·휴일/휴무 결재 워크플로우를 모두 제외**했으므로 본 용어는 (deprecated) 서술의 이해를 위한 정의로만 남긴다 |
| **FW / MW / SM** | 초안 프로세스 코드 표기법. FW=일반사용자 프로세스, MW=매니저 프로세스, SM=관리자 프로세스 (추적성 확보 위해 유지) |

> **표기법 유지 원칙**: 원본 초안의 프로세스 코드(`FW#`, `M#`, `S#`)를 그대로 보존하여
> 초안 ↔ 본 문서 간 추적성을 확보한다. 초안의 명백한 오타(STMP, FW4 중복, F5/F6)는
> 교정하되 원래 의도는 보존한다(11장 참조).

## 2. 시스템 개요 & 범위(Scope)

### 2.1 한 줄 정의
**휴일을 포함한 전(全)일자에 대해 30분 단위로 운영되는 자동차 세차 예약 서비스.**
매장은 다중 베이(A1~AN)를 보유하여 동일 시간대에 베이 수만큼의 사용자를 동시 수용한다.

### 2.2 MVP 범위 정의
| 구분 | 대상 | 처리 방침 | 포함 기능 |
|------|------|-----------|-----------|
| **1차 (구현)** | FO 사용자 플로우 | 이번 MVP에서 실제 구현 | 로그인, 매장/매니저/슬롯 선택 예약, 세차완료/예약취소, (확장) 후기 작성 |
| **2차 (문서화)** | 매니저·관리자 BO | 이번에는 **문서화만**, 2차 과제로 이관 | 예약 대행(+세차완료/취소 보조), 매장별 관리·매출 확인 |
| **추후 (별도 과제)** | 인프라/확장 | 로드맵에만 기재 | 카드결제, SNS 인증, 실시간 알림 |

> **⚠️ 구현-요구 정합성 이행 메모 (v1.6)**
> 2차 백엔드 구현 산출물(Phase 5~7)에 포함된 **결재(승인) 워크플로우·STORE_ADMIN(매장 최고권한 매니저) 역할·M6(매장예약 승인)·M7(매니저 가입 승인)** 등은
> v1.6 단순화 요구(3역할 / 승인 분기 제거 / 휴일·휴무 도메인 데이터화)를 **초과하는 범위**이므로 **후속 정리 대상**이다.
> 반면 **매니저 예약 대행(M3) + 세차완료 보조(M4) / 취소 보조(M5)** 는 v1.6 요구에 부합하므로 **유지**한다.
> (현 구현이 본 정의서를 앞서간 부분으로, 코드 정리는 별도 후속 작업으로 추적한다.)

### 2.3 Out-of-Scope (명시적 제외)
이번 MVP에서 **구현하지 않는** 항목을 명확히 한다.
- **카드결제 / PG 연동**: 결제는 현장결제만. 카드결제는 추후 과제.
- **SNS 소셜 로그인**: 이메일(SMTP) 인증만. SNS 인증은 추후 과제.
- **실시간 알림 (메일/푸시)**: 알림 정책 미확정. 미해결 이슈로 분류(13장).
- **BO(매니저/관리자) 기능 구현**: 2차 과제. 본 문서에 설계만 기재.

> ※ (v1.2) 기존 Out-of-Scope였던 "Nuxt.js 전환"은 **Nuxt 4 채택으로 확정·반영**되어 더 이상 제외 항목이 아니다 (12.1 참조).

## 3. 사용자 권한(Role) 정의

### 3.1 역할 구성
> *(v1.6)* 역할 체계를 **사용자/매니저/관리자 3역할**로 단순화한다. 기존 **매장 최고권한 매니저(STORE_ADMIN)** 역할은 **제거**한다(승인·재상신 권한 자체가 사라지므로 별도 상위 매니저 역할이 불필요).

| 역할 | 설명 |
|------|------|
| **사용자** (User) | 매장 예약 서비스를 사용하는 고객 |
| **매니저** (Manager) | 매장마다 N명 존재. 사용자 예약을 **대행**하고 대행 예약의 **세차완료/취소를 보조** |
| **관리자** (Admin) | 매장별 예약자/사용자/후기·평점/회원정보/매출을 총괄 관리 |

### 3.2 권한별 액션 매트릭스
> *(v1.6)* 매트릭스를 **3역할(사용자/매니저/관리자)** 기준으로 재작성한다. 제외 항목(매장 예약 승인 M6, 매니저 가입 승인 M7, 사용자 매장가입 승인, 매니저 휴무 1차 승인, 매장 휴일/휴무 최종 승인 등 결재·승인 액션)은 **삭제**되었다.

| 액션 | 사용자 | 매니저 | 관리자 |
|------|:------:|:------:|:------:|
| 회원가입/로그인 | O | O | O |
| 매장 예약 (본인, FW5) | O | - | - |
| 세차완료 (FW6) | O | - | - |
| 예약취소 (FW7) | O | - | - |
| 후기 작성 (§9) | O | - | - |
| 사용자 예약 대행 (M3) | - | O | - |
| 대행 예약 세차완료 보조 (M4) | - | O | - |
| 대행 예약 취소 보조 (M5) | - | O | - |
| 매장별 예약자 관리 (S4) | - | - | O |
| 매장별 사용자 관리 (S5) | - | - | O |
| 매장별 후기/평점 확인 (S6) | - | - | O |
| 회원정보 확인 (S7) | - | - | O |

## 4. 회원가입 & 인증

### 4.1 가입 페이지 구성
사용자 / 매니저 / 관리자 **각각의 가입 페이지 3종**을 별도로 구성한다.

### 4.2 인증과 승인의 분리 (초안 모순 해소)
> **원본 초안의 모순**: "모든 가입은 인증이 필요하다(승인은 필요하지 않다)"고 명시했으나,
> 별도로 "매니저 가입은 관리자 승인 필요", "사용자 매장가입은 매장 최고관리자 승인 필요"라고
> 기술되어 충돌한다. **v1.6에서는 이 모순을 "이메일 인증만으로 활성, 승인 분기 제거" 방향으로 최종 해소한다.**
> 즉 가입 승인(역할/매장가입 승인) 단계를 전면 제거하고, **모든 가입자는 이메일 인증 완료 시 곧바로 `ACTIVE`** 가 된다.

| 구분 | 정의 | 적용 대상 |
|------|------|-----------|
| **이메일 인증 (필수)** | SMTP 기반 이메일 본인확인. 추후 SNS 인증 추가 예정. | **모든 가입자 공통** |

즉, 모든 가입은 이메일 인증만 거치면 활성화되며,
**별도 승인 단계는 두지 않는다** *(v1.6: 승인 분기 제거)*.

### 4.3 가입 흐름 요약 *(v1.6: 승인 분기 제거)*
> *(v1.6)* **모든 가입 = 이메일 인증 → `ACTIVE` 로 단일화**한다. 가입 유형(사용자/매니저/관리자)에 관계없이 승인 주체·승인 단계는 없다.

> **(deprecated — v1.6 보존)** 아래 표는 승인 분기가 있던 v1.5 이전 흐름이다. 추적성 위해 보존하되 **현재는 적용하지 않는다**.
>
> | 가입 유형 | 프로세스 | 인증 | 승인 주체 |
> |-----------|----------|------|-----------|
> | 일반 사용자 가입 | FW1 → S3 | 이메일 인증 필수 | (가입 자체는 인증으로 충분) |
> | 사용자 매장가입 | - | 이메일 인증 필수 | 매장 최고권한 매니저 |
> | 최고 매니저 가입 | M1 → S3 | 이메일 인증 필수 | 관리자 |
> | 일반 매니저 가입 | M1 → M7 | 이메일 인증 필수 | 매장 최고권한 매니저(M7) |

### 4.4 회원가입 상태 머신 *(v1.6: 3상태 단순화)*
> *(v1.6)* `PENDING_APPROVAL`·`REJECTED` 상태를 제거하고 **`REQUESTED → EMAIL_VERIFIED → ACTIVE`** 3상태로 단순화한다.

| 상태 | 설명 | 다음 상태 | 전이 조건 |
|------|------|-----------|-----------|
| `REQUESTED` (가입신청) | 가입 폼 제출 완료 | `EMAIL_VERIFIED` | 인증 메일 발송 |
| `EMAIL_VERIFIED` (이메일인증) | 인증 링크 클릭 완료 | `ACTIVE` | 인증 완료 즉시 활성 |
| `ACTIVE` (활성) | 정상 이용 가능 | - | - |

> 모든 가입은 `EMAIL_VERIFIED` → `ACTIVE`로 바로 전이된다(승인 분기 없음).

## 5. 매장·베이·슬롯 도메인 모델

### 5.1 매장 (Store)
- 한 매장은 **N개의 베이**(A1 ~ AN)를 보유한다.
- 운영 시간: **기본 24시간 / 365일**. 단, **매장 휴일 설정** 및 **매니저 휴무**가 존재한다.
  - 매장 운영 시간(24시간)은 변경하지 않는다. **매니저 근무는 3교대로 운영**되며(5.5 참조), 매니저 휴무는 전일 또는 교대조 단위로 발생할 수 있다.

### 5.2 슬롯 (Slot) 정의
> **슬롯 = (매장, 베이, 날짜, 30분 시간단위)** 의 조합. 이 조합은 시스템 전체에서 유일(UNIQUE)하다.

- 시간 단위: **30분** 고정.
- 동일 시간대 최대 수용 인원 = **해당 매장의 베이 수(N)**.
  - 예: 베이가 A1~A3(3개)인 매장은 09:00~09:30 시간대에 최대 3명 예약 가능.

> **베이 슬롯 점유 시 선택 불가 (UX 규칙)**: 슬롯 UNIQUE 제약을 사용자 화면 규칙으로 명문화한다.
> 특정 베이가 특정 `(날짜, 시간)` 슬롯에서 이미 **예약(RESERVED)** 또는 **세차완료(COMPLETED)** 상태이면,
> 그 슬롯에서 해당 베이는 **선택 불가(비활성)** 로 표시되어야 한다. 슬롯 status 전이 및 동시성 처리는 7장과 연계한다.

### 5.3 엔티티 관계 개략도
```
사용자(User) ──< 예약(Reservation) >── 슬롯(Slot)
                      │                     │
                      │                  베이(Bay) ──< 매장(Store)
                      │                                  │
매니저(Manager) ──────┤                          매장 휴일(StoreHoliday)
   │                  │
매니저 휴무(ManagerDayoff)                       가격(Price: 차종 × 서비스)
                      │
                      └── 후기/평점(Review) >── 사용자 / 매장 / 매니저
```

### 5.4 주요 엔티티 요약
| 엔티티 | 핵심 속성(개략) | 관계 |
|--------|----------------|------|
| User | id, email, role, 가입상태 | Reservation 1:N |
| Store | id, name, bayCount | Bay 1:N, Manager 1:N |
| Bay | id, storeId, code(A1~AN) | Slot 1:N |
| Manager | id, storeId | Reservation(대행) 1:N. *(v1.6: `isStoreAdmin` 속성 제거)* |
| Slot | storeId, bayId, date, timeSlot (UNIQUE) | Reservation 1:1 |
| Reservation | id, userId, slotId, status, 결제방식 | - |
| Review | id, userId, storeId, managerId, rating(1~5), text | 예약완료 사용자만 작성 |
| Price | carType, serviceType, amount | 차종 × 서비스 매트릭스 |
| StoreHoliday | storeId, date | 도메인 데이터(등록 주체 추후 확정/관리자 관리). 슬롯 가용성에만 반영(§6.1). *(v1.6: 승인상태 제거)* |
| ManagerDayoff | managerId, date, **휴무유형(FULL_DAY / SHIFT_1·2·3)** | 도메인 데이터(등록 주체 추후 확정/관리자 관리). 휴무유형은 전일 또는 교대조 단위(5.5 참조). *(v1.6: 승인상태 제거)* |

### 5.5 매니저 근무 교대(3교대) & 휴무 유형 *(v1.3 추가)*

매장 운영은 **24시간 유지**(5.1 변경 없음)하되, 매니저 근무는 **3교대** 기준으로 운영한다.

#### 5.5.1 교대 정의
| 교대 | 코드 | 시간대 |
|------|------|--------|
| 1교대(오전조) | `SHIFT_1` | 06:00 ~ 14:00 |
| 2교대(오후조) | `SHIFT_2` | 14:00 ~ 22:00 |
| 3교대(야간조) | `SHIFT_3` | 22:00 ~ 06:00(익일) |

#### 5.5.2 휴무 유형
> ※ 초안에서 처음 언급된 '오전/오후 반차' 개념을 **3교대 기반 부분 휴무로 일반화**한 것이다.

| 휴무 유형 | 코드 | 의미 | 예약 영향 |
|-----------|------|------|-----------|
| 전일 휴무 | `FULL_DAY` | 해당 날짜 종일 휴무 | 그날 **전체 슬롯 비활성** |
| 교대조 단위 휴무 | `SHIFT_1` / `SHIFT_2` / `SHIFT_3` (1개 이상) | 특정 교대 시간대만 휴무 | 해당 교대 시간대의 **30분 슬롯만 비활성**, 그 외 시간대는 예약 가능 |

- 한 날짜에 교대조 휴무를 **복수 지정**할 수 있다(예: `SHIFT_1` + `SHIFT_2`).
- 휴무 유형은 ManagerDayoff 엔티티의 속성으로 보관한다(5.4 참조).

## 6. 예약 기능 (FO 핵심 — 1차 구현)

### 6.1 사용자 예약
- **가입/승인된 매장만** 예약 가능.
- **매니저 휴무(전일/교대조) 시간대의 슬롯은 예약 불가** (해당 매니저 지정 시 휴무 슬롯 비활성).
  - 전일 휴무(`FULL_DAY`)면 그날 전체 슬롯이 비활성, 교대조 휴무(`SHIFT_1`/`SHIFT_2`/`SHIFT_3`)면 해당 교대 시간대의 30분 슬롯만 비활성된다(5.5 참조).
- **베이 슬롯 점유 시 선택 불가**: 특정 베이가 특정 `(날짜, 시간)` 슬롯에 이미 예약(RESERVED)/세차완료(COMPLETED)이면 그 슬롯에서 해당 베이는 비활성으로 표시한다(5.2 정의 및 7장 동시성 연계).
  - 선택 순서는 **매장→매니저→차종→서비스(1페이지) → 날짜·시간 선택 → 그 시간대 베이 그리드에서 점유 베이 비활성·가용 베이 버튼 선택(2페이지)** 이다 *(v1.4 갱신)*. 베이 점유 충돌은 날짜·시간이 정해진 뒤 판정 가능하므로, **날짜·시간 선택 후 해당 시간대의 점유된 베이를 비활성**으로 표시한다(슬롯 그리드 UX). 단계별 화면 분할은 6.5 참조.

### 6.2 매니저 예약 대행
- 로그인한 매니저의 **소속 매장 정보 기준**으로 사용자 예약을 대행한다.
- 매니저 본인의 **휴무를 반영**하여 예약 가능 슬롯을 제한한다.

### 6.3 검색 UX 요건
- 사용자 검색 / 매니저 검색은 **select 박스**로 제공한다.
- select 박스에 **text 입력 시 필터 검색**(자동 필터링)이 동작해야 한다.

### 6.4 결제
- **현장결제만** 지원 (예약 시점에는 결제 미수행, 매장 방문 시 결제).
- 카드결제는 추후 과제 (Out-of-Scope, 2.3 참조).

### 6.5 예약 화면 구성 (3단계 위저드) *(v1.4 추가)*

> 이해관계자 결정으로 예약 플로우를 **3개 화면(라우트)으로 분할한 위저드**로 구성한다.
> 단계 간 진행 데이터는 **클라이언트 상태(Pinia 스토어)** 에 보관하며, 1단계 구현에서는
> **in-memory Pinia만** 사용한다(데이터 진화 1단계, 12.3 연계).

#### 6.5.1 단계별 화면 정의
| 단계 | 라우트 | 화면 역할 | 선택/입력 항목 | 다음 액션 |
|------|--------|-----------|----------------|-----------|
| 1페이지 | `/reserve` | 예약 기본 정보 선택 | 매장 → 매니저 → 차종 → 서비스 선택 (서비스 선택 시 **가격 자동 표시**, 10장 매트릭스) | 4개 선택 완료 시 **"다음"** |
| 2페이지 | `/reserve/slot` | 슬롯·베이 선택 | 날짜·시간 선택 → **해당 시간대 베이 점유 그리드**에서 가용 베이 버튼 선택 | **"예약하기"** 로 확정 (충돌 시 재선택 토스트) |
| 3페이지 | `/reserve/done` | 예약 완료 | 예약 완료 요약 표시 | - |

#### 6.5.2 베이 선택 방식 (단일화)
- 베이 선택은 **버튼 그리드 단일 방식**으로 한다.
- 기존 검색 select box(SearchableSelect) 방식의 베이 선택은 **폐지**한다(중복 선택 제거).
- 베이 점유(RESERVED/COMPLETED) 비활성 규칙(5.2·6.1)은 그대로 유지하며, 점유 베이는
  그리드에서 비활성, 가용 베이만 선택 가능한 버튼으로 노출한다.

> ※ 매장·매니저 검색 UX(6.3, select 박스 + text 필터)는 그대로 유지된다.
> 단일화 대상은 **베이 선택**에 한한다.

#### 6.5.3 진행 데이터 영속성 & 진입 가드
- 1단계 구현은 **in-memory Pinia만** 사용한다 → **브라우저 새로고침 시 진행 데이터는 초기화**된다.
- 미완료 단계로 직접 진입(예: 1페이지를 거치지 않고 `/reserve/slot` 진입)하는 경우
  **1페이지(`/reserve`)로 되돌리는 가드**를 둔다.
- 추후 서버/Redis/JWT 기반 진행상태 영속화는 데이터 진화 2·3단계 과제로 이관한다(12.3 연계).

## 7. 동시 예약(동시성) 처리 — 권장안

> **배경**: 초안에서 개발자가 직접 "예약이 중복될 경우 데이터 동기화 문제 해결방법?"을 질문했다.
> 데이터 진화 3단계(12장)에 맞춰 단계별 권장안을 제시한다. 최종 목표는 3단계의 DB 유니크 제약이며,
> 1·2단계는 그에 도달하기 위한 점진적 시뮬레이션이다.

### 7.1 단계별 권장안
| 단계 | 데이터 계층 | 권장 기법 | 핵심 동작 |
|------|-------------|-----------|-----------|
| **1단계** | 프론트 더미 | 클라이언트 슬롯 잠금 시뮬레이션 + 낙관적 갱신 | 슬롯 선택 시 즉시 `status='예약중'` 표시. 확정 직전 클라이언트 상태로 충돌 감지 후 재선택 유도 |
| **2단계** | Spring Boot 인메모리 더미 | 슬롯 UNIQUE 제약(매장+베이+날짜+시간) + 락 | **MyBatis** 기반 낙관적 락(version 컬럼 비교 UPDATE) 또는 비관적 락(`SELECT ... FOR UPDATE`) 적용. 동시 요청 시 한 건만 성공 |
| **3단계** | MySQL | 트랜잭션 + 유니크 인덱스 | DB 레벨에서 중복 INSERT를 원천 차단. 충돌 시 사용자에게 재선택 유도 |

### 7.2 단계별 장단점 (1줄 비교)
| 단계 | 장점 | 단점 |
|------|------|------|
| 1단계 | 백엔드 없이 UX 검증 가능 | 클라이언트 신뢰 불가 — 진짜 동시성 보장 못 함 |
| 2단계 | 서버 검증 도입, 실제 락 동작 검증 | 인메모리라 재시작 시 휘발, 분산환경 미고려 |
| 3단계 | DB 유니크 인덱스로 무결성 원천 보장 | 충돌 처리(재선택 유도) UX 필수 구현 |

### 7.3 2단계 락 전략 비교
| 기법 | 적합 상황 | 장점 | 단점 |
|------|-----------|------|------|
| 낙관적 락 (version 컬럼) | 충돌 빈도 낮음 | 락 대기 없음, 처리량 높음 | 충돌 시 재시도 로직 필요. **MyBatis는 `UPDATE ... SET version=version+1 WHERE id=? AND version=?` 의 영향 행 수(0=충돌)로 판정** |
| 비관적 락 (SELECT ... FOR UPDATE) | 충돌 빈도 높음(인기 슬롯) | 충돌 자체를 차단 | 락 대기/데드락 위험, 처리량 저하. **MyBatis 매퍼에 `FOR UPDATE` SQL 직접 작성** |

> **권장**: 슬롯 `UNIQUE(storeId, bayId, date, timeSlot)` 제약을 최종 방어선으로 두고,
> 일반 슬롯은 낙관적 락, 경합이 잦은 슬롯은 비관적 락을 선택적으로 적용한다.

> **UX 연계(6.1)**: 슬롯이 `RESERVED`/`COMPLETED` 상태로 전이되면, 해당 `(날짜, 시간)`에서 그 베이는
> 선택 불가(비활성)로 노출되어야 한다(6.1 "베이 슬롯 점유 시 선택 불가" 규칙). 위 동시성 처리는 확정 시점의
> 충돌을 방지하고, 본 UX는 선택 단계에서 점유 베이를 사전 차단하여 충돌 빈도 자체를 낮춘다.

## 8. 휴일/휴무 (도메인 데이터) *(v1.6: 결재 워크플로우 제외)*

> *(v1.6)* 휴일/휴무는 **2단계 승인 결재 워크플로우 없이 도메인 데이터로만 유지**한다.
> 즉 `StoreHoliday`·`ManagerDayoff` 는 단순 데이터로 등록·관리되며, 그 효과는 오직 **슬롯 가용성(§6.1)** 에 반영된다
> (전일 휴무/교대조 휴무가 해당 슬롯을 비활성화). **2단계 승인 결재(매장 최고매니저 → 관리자)는 제거**한다.
> **등록 주체**는 추후 확정한다(관리자 관리로 운영 가정, §13.2 참조). 휴무 유형(전일/교대조) 정의는 §5.5에 그대로 유지된다.

---

> **(deprecated — v1.6) 아래 §8.1~§8.3 의 결재 서술·상태 머신은 추적성 위해 보존하되 현재는 적용하지 않는다.**

### 8.1 매장 휴일 *(deprecated — v1.6)*
- 매니저가 **매장 휴일 설정 결재**를 올리면 **관리자가 승인**한다.

### 8.2 매니저 휴무 *(deprecated — v1.6)*
- 매니저가 **휴무 결재**를 올리면 → **매장 최고권한 매니저가 1차 승인** →
  최고권한 매니저가 **관리자에게 재상신**하여 최종 승인을 요청한다.
- **우선 적용 방식**: 별도 워크플로우 엔진 없이 **상태값 변경 방식**으로 구현한다.
- **부분 휴무(교대조 단위)도 동일 결재 흐름을 따른다** — 전일/교대조 구분 없이 신청 → 최고매니저 1차 승인 → 관리자 재상신·최종 승인으로 동일하게 처리한다(휴무 유형 정의는 5.5 참조).

### 8.3 결재 상태 머신 *(deprecated — v1.6)*
| 상태 | 설명 | 다음 상태 | 전이 주체 |
|------|------|-----------|-----------|
| `SUBMITTED` (신청) | 결재 상신 | `APPROVED_L1` / `REJECTED` | 신청자 → (검토자) |
| `APPROVED_L1` (1차승인) | 매장 최고매니저 승인 | `APPROVED_L2` / `REJECTED` | 매장 최고권한 매니저 |
| `APPROVED_L2` (2차승인/확정) | 관리자 최종 승인 | `CONFIRMED` | 관리자 |
| `CONFIRMED` (확정) | 휴일/휴무 확정 반영 | - | - |
| `REJECTED` (반려) | 어느 단계든 거부 | `SUBMITTED` (재신청) | 검토자 |

> **매장 휴일**은 매니저 신청 → 관리자 승인의 단일 승인(L1 생략 또는 L1=관리자)으로 단순화 가능하다.
> **매니저 휴무**는 위 표의 2단계(최고매니저 → 관리자) 흐름을 따른다.

## 10. 가격 정책 (표준안)

## 11. 프로세스 정의 (초안 FW/MW/SM 정규화)

> **오타 교정 원칙**: 초안의 `FW4` 중복(매니저선택/매장예약), `F5`/`F6` 표기는 아래와 같이 정규화한다.
> 원래 프로세스 의도는 보존한다.

### 11.1 프로세스 코드 정의 (정규화)
| 영역 | 코드 | 단계 |
|------|------|------|
| 일반사용자(FW) | FW1 | 회원가입 |
| | FW2 | 로그인 |
| | FW3 | 매장선택 |
| | FW4 | 매니저선택 |
| | **FW5** | 매장예약 *(초안 FW4 중복 → FW5로 교정)* |
| | **FW6** | 세차완료 *(초안 F5 → FW6으로 교정)* |
| | **FW7** | 예약취소 *(초안 F6 → FW7로 교정)* |
| 매니저(M) | M1 | 회원가입 |
| | M2 | 로그인 |
| | M3 | 매장예약(대행) |
| | M4 | 세차완료 |
| | M5 | 예약취소 |
| | M6 | 매장예약 승인 *(deprecated/제외 — v1.6)* |
| | M7 | 매니저 가입 승인 *(deprecated/제외 — v1.6)* |
| 관리자(S) | S1 | 회원가입 |
| | S2 | 로그인 |
| | S3 | 가입승인 *(deprecated/제외 — v1.6: 승인 분기 제거)* |
| | S4 | 매장별 예약자 관리 |
| | S5 | 매장별 사용자 관리 |
| | S6 | 매장별 후기/평점 확인 |
| | S7 | 회원정보 확인 |
| | S8 | 매출 확인 |

> **교정 매핑 추적표**
> | 초안 표기 | 교정 후 | 사유 |
> |-----------|---------|------|
> | STMP | SMTP | 오타 |
> | FW4(매장예약) | FW5 | FW4(매니저선택)와 번호 중복 |
> | F5(세차완료) | FW6 | 접두어 누락 + 번호 재배치 |
> | F6(예약취소) | FW7 | 접두어 누락 + 번호 재배치 |

### 11.2 가입 프로세스 *(v1.6: 승인 단계 제거 → 단순화)*
> *(v1.6)* 승인 시퀀스(→S3, →M7)를 제거하고 매니저를 단일화한다. **모든 가입은 이메일 인증 필수(승인 없음)**.

| 가입 유형 | 시퀀스 | 비고 |
|-----------|--------|------|
| 일반 사용자 | FW1 | 이메일 인증 필수(승인 없음) |
| 매니저 | M1 | 이메일 인증 필수(승인 없음) |
| 관리자 | S1 | 이메일 인증 필수(승인 없음) |

### 11.3 업무 프로세스
**(a) 세차완료 케이스**
```
FW2(로그인) → FW3(매장선택) → FW5(매장예약) → FW6(세차완료)
```
> 초안 `FW2 → FW3 → M6 → FW4 → FW5`를 정규화. *(v1.6: 매장예약 승인 M6 제거)* 매장예약은 FW5, 세차완료는 FW6.

**(b) 예약 후 취소**
```
FW2(로그인) → FW3(매장선택) → FW5(매장예약) → FW7(예약취소)
```
> 초안 `FW2 → FW3 → M6 → FW6`을 정규화. *(v1.6: 매장예약 승인 M6 제거)* 취소=FW7.

**(c) 예약취소 케이스 2 (승인 전 취소)**
```
FW2(로그인) → FW3(매장선택) → FW7(예약취소)
```
> 초안 `FW2 → FW3 → FW6`을 정규화 (취소=FW7).

**(d) 기존 사용자**
- 최초 사용자 플로우에서 **FW1(회원가입)을 생략**하고 진행.
