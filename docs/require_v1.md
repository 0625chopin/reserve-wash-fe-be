> **문서 버전**: v1.15

## 1. 문서 개요

### 1.1 문서 목적
개발자가 작성한 원시 요구사항 초안을 정식 요구사항 정의서로 재편성하여,
기획·개발·이해관계자가 동일한 기준으로 MVP 범위와 동작을 이해하고
의사결정에 활용할 수 있도록 한다.

### 1.3 용어 정의
| 용어 | 정의 |
|------|------|
| **FO** (Front-Office) | 사용자(고객)가 직접 사용하는 화면/플로우. 로그인·예약 페이지 등 |
| **BO** (Back-Office) | 매니저·관리자가 사용하는 관리 화면/플로우. *(v1.7)* **일반매장매니저/매장매니저관리자/관리자 각 역할별로 화면을 분리**하며, 일반매장매니저는 예약 대행·보조 및 휴가/반차 신청, 매장매니저관리자는 매니저 기능 전부 + 가입 1차 승인 + 휴가/반차 승인, 관리자는 가입 2차 최종 승인·매출·예약상태·매니저 근무상태 확인을 담당한다. ~~**(v1.6 범위 축소)** 매니저는 사용자 예약 대행(+세차완료/취소 보조), 관리자는 매장별 관리·매출 확인으로 한정(결재/승인 제외)~~ *(deprecated — v1.6)* |
| **매장** (Store) | 세차 서비스를 제공하는 물리적 지점. N개의 베이를 보유 |
| **베이** (Bay) | 매장 내 개별 세차 라인. `A1 ~ AN`으로 식별. 동일 시간대 베이 수만큼 동시 예약 수용 가능 |
| **슬롯** (Slot) | 예약 가능 단위. `(매장, 베이, 날짜, 30분 시간단위)` = 1슬롯 |
| **일반사용자** (`USER`) *(v1.7)* | 매장 예약 서비스를 사용하는 고객. 일반 예약 이벤트(예약/세차완료/취소/후기)만 수행. 회원가입은 이메일 인증 즉시 자동 활성(승인 분기 없음) |
| **일반매장매니저** (`MANAGER`) *(v1.7)* | 매장마다 N명 존재. 사용자 예약을 **대행·보조**(예약취소·완료)하고 **휴가/반차를 신청**한다. 사용자와 페이지가 분리됨 |
| **매장매니저관리자** (`STORE_ADMIN`) *(v1.7)* | 일반매장매니저 기능 전부 + **일반매장매니저 가입 1차 승인** + **일반매장매니저 휴가/반차 승인**. v1.6에서 제거했던 상위 매니저 역할을 부활시킨 것 |
| **관리자** (`ADMIN`) *(v1.7)* | **매니저 가입 최종(2차) 승인** + 매출·매장 예약상태(예약/취소/완료)·매장별 매니저 근무상태 확인 |
| **결재 (= 승인)** *(v1.7 부활)* | 신청 건을 상위 권한자가 검토·확정하는 워크플로우. 본 문서에서 "결재"와 "승인"은 동의어로 사용. **가입 승인은 2단계**(매장매니저관리자 1차 → 관리자 2차), **휴가/반차 승인은 1단계**(매장매니저관리자 종결)로 단계 수가 다르다. ~~v1.6에서 모든 결재 워크플로우 제외~~ *(deprecated — v1.6)* |

| **FW / MW / SM** | 초안 프로세스 코드 표기법. FW=일반사용자 프로세스, MW=매니저 프로세스, SM=관리자 프로세스 (추적성 확보 위해 유지) |

> **표기법 유지 원칙**: 원본 초안의 프로세스 코드(`FW#`, `M#`, `S#`)를 그대로 보존하여
> 초안 ↔ 본 문서 간 추적성을 확보한다. 초안의 명백한 오타(STMP, FW4 중복, F5/F6)는
> 교정하되 원래 의도는 보존한다(11장 참조).

## 2. 시스템 개요 & 범위(Scope)

### 2.1 한 줄 정의
**휴일을 포함한 전(全)일자에 대해 30분 단위로 운영되는 자동차 세차 예약 서비스.**
매장은 다중 베이(A1~AN)를 보유하여 동일 시간대에 베이 수만큼의 사용자를 동시 수용한다.

### 2.2 MVP 범위 정의
| 구분 | 대상 | 처리 방침 | 포함 기능 |
|------|------|-----------|-----------|
| **1차 (구현)** | FO 일반사용자 플로우 | 이번 MVP에서 실제 구현 | 로그인, 매장/매니저/슬롯 선택 예약, 세차완료/예약취소, (확장) 후기 작성 |
| **2차 (문서화·역할별 분리)** *(v1.7)* | BO 3역할(일반매장매니저·매장매니저관리자·관리자) | 이번에는 **문서화·화면 설계만**, 2차 과제로 이관 | **역할별로 분리된 BO 화면**(§12.4): 일반매장매니저=예약 대행·보조 + 휴가/반차 신청 / 매장매니저관리자=매니저 기능 + 가입 1차 승인 + 휴가/반차 승인 / 관리자=가입 2차 최종 승인 + 매출·예약상태·매니저 근무상태 확인 |
| **추후 (별도 과제)** | 인프라/확장 | 로드맵에만 기재 | 카드결제, SNS 인증, 실시간 알림 |

> **⚠️ 구현-요구 정합성 이행 메모 *(v1.7 갱신)***
> v1.6에서 "코드가 정의서를 앞서간 초과 범위 → 후속 정리 대상"으로 보았던 항목들이,
> v1.7 요구 번복(4역할 + 2단계 가입 승인 부활)으로 **다시 요구에 부합**하게 되었다. 즉:
> - **STORE_ADMIN(매장매니저관리자) 역할**, **`ApprovalStatus`(`SUBMITTED→APPROVED_L1→CONFIRMED→REJECTED`) 승인 상태**, **매니저 가입 2단계 승인** → v1.7 요구로 정당화되어 **유지**한다(코드 제거 불필요).
> - **매니저 예약 대행(M3) + 세차완료 보조(M4) / 취소 보조(M5)** → 계속 부합, **유지**.
> - 휴가/반차 승인은 **1단계**(매장매니저관리자 종결)이므로, 만약 구현이 휴가/반차를 2단계로 처리한다면 그 부분만 1단계로 재조정 대상이다.
>
> ~~**(deprecated — v1.6)** 위 결재·STORE_ADMIN·M6/M7을 초과 범위로 보고 후속 정리 대상으로 분류~~ (v1.7에서 번복).

### 2.3 Out-of-Scope (명시적 제외)
이번 MVP에서 **구현하지 않는** 항목을 명확히 한다.
- **카드결제 / PG 연동**: 결제는 현장결제만. 카드결제는 추후 과제.
- **SNS 소셜 로그인**: 이메일(SMTP) 인증만. SNS 인증은 추후 과제.
- **실시간 알림 (메일/푸시)**: 알림 정책 미확정. 미해결 이슈로 분류(13장).
- **BO 기능 구현**: 2차 과제. 본 문서에 설계만 기재. *(v1.7)* 단, BO 화면은 **일반매장매니저·매장매니저관리자·관리자 3역할별로 분리**하여 설계한다(§12.4). 가입 승인(2단계)·휴가/반차 승인(1단계) 워크플로우는 **구현 범위에 포함**(설계·구현 모두 필요)이며 Out-of-Scope가 아니다.

> ※ (v1.2) 기존 Out-of-Scope였던 "Nuxt.js 전환"은 **Nuxt 4 채택으로 확정·반영**되어 더 이상 제외 항목이 아니다 (12.1 참조).

## 4. 회원가입 & 인증

### 4.1 가입 페이지 구성
*(v1.7)* 일반사용자(`USER`) / 일반매장매니저(`MANAGER`) / 매장매니저관리자(`STORE_ADMIN`) / 관리자(`ADMIN`) **각각의 가입 페이지를 역할별로 별도 구성**한다.

*(v1.9)* **로그인 페이지도 역할군별로 분리**한다 — 일반사용자(`/login`, 기존 유지) · 매니저 계열(`/manager/login`: 일반매장매니저+매장매니저관리자 공용) · 관리자(`/admin/login`). 각 페이지는 해당 역할군의 (개발용) 빠른 로그인을 제공한다. **매니저 회원가입(`/manager/signup`)** 은 매니저 로그인 페이지에서 진입하며, **소속 매장을 선택**하고 `role=MANAGER`로 신청하면 가입 즉시 `PENDING_APPROVAL_L1` 상태로 등록(자동 로그인 없음)되어 **2단계 승인(M7 매장매니저관리자 1차 → S3 관리자 2차) 후 `ACTIVE`** 가 된다(§4.4). 관리자 회원가입 폼은 두지 않는다(시드/내부 생성).

### 4.2 인증과 승인의 분리 (초안 모순 해소) *(v1.7: 역할별 승인 분기 부활)*
> **원본 초안의 모순**: "모든 가입은 인증이 필요하다(승인은 필요하지 않다)"고 명시했으나,
> 별도로 "매니저 가입은 관리자 승인 필요", "사용자 매장가입은 매장 최고관리자 승인 필요"라고
> 기술되어 충돌한다.
>
> *(v1.7)* 이 모순을 **"역할에 따라 인증과 승인을 분리 적용"** 방향으로 해소한다.
> - **`USER`**: 이메일 인증 완료 즉시 `ACTIVE`(승인 분기 없음, 자동 가입).
> - **`MANAGER`·`STORE_ADMIN`**: 이메일 인증 + **2단계 승인**(매장매니저관리자 1차 → 관리자 2차)을 모두 통과해야 `ACTIVE`. **승인 완료(`ACTIVE`) 이전에는 로그인 불가**.
>
> ~~**v1.6에서는 이 모순을 "이메일 인증만으로 활성, 승인 분기 제거" 방향으로 최종 해소한다.** 즉 가입 승인 단계를 전면 제거하고, 모든 가입자는 이메일 인증 완료 시 곧바로 `ACTIVE`가 된다.~~ *(deprecated — v1.6)*

| 구분 | 정의 | 적용 대상 |
|------|------|-----------|
| **이메일 인증 (필수)** | SMTP 기반 이메일 본인확인. 추후 SNS 인증 추가 예정. | **모든 가입자 공통** |
| **2단계 가입 승인 (필수)** *(v1.7)* | 매장매니저관리자 1차 승인 → 관리자 2차 최종 승인. 승인 완료 전 로그인 불가. | **`MANAGER`·`STORE_ADMIN` 가입자** |

즉, *(v1.7)* `USER`는 이메일 인증만으로 활성화되지만,
**`MANAGER`·`STORE_ADMIN`은 이메일 인증 후 2단계 승인까지 완료해야 `ACTIVE`** 가 되고 그 전에는 로그인할 수 없다.

### 4.4 회원가입 상태 머신 *(v1.7: 2단계 승인 상태 부활)*
> *(v1.7)* v1.6에서 제거했던 `PENDING_APPROVAL` 계열·`REJECTED` 상태를 **부활**시키고, 2단계 승인을 표현하도록 **`REQUESTED → EMAIL_VERIFIED → PENDING_APPROVAL_L1 → PENDING_APPROVAL_L2 → ACTIVE`** (+ `REJECTED`)로 확장한다.
> **오직 `ACTIVE` 상태만 로그인 가능**하며, 그 이전 상태(`REQUESTED`·`EMAIL_VERIFIED`·`PENDING_APPROVAL_L1`·`PENDING_APPROVAL_L2`·`REJECTED`)에서는 **로그인 불가**다.

| 상태 | 설명 | 다음 상태 | 전이 조건 / 전이 주체 |
|------|------|-----------|----------------------|
| `REQUESTED` (가입신청) | 가입 폼 제출 완료 | `EMAIL_VERIFIED` | 인증 메일 발송 |
| `EMAIL_VERIFIED` (이메일인증) | 인증 링크 클릭 완료 | **`USER`: `ACTIVE` 직행** / **`MANAGER`·`STORE_ADMIN`: `PENDING_APPROVAL_L1`** | `USER`는 인증 즉시 활성(승인 없음). 매니저 계열은 1차 승인 대기로 진입 |
| `PENDING_APPROVAL_L1` (1차 승인 대기) *(v1.7)* | 매장매니저관리자 1차 승인 대기 | `PENDING_APPROVAL_L2` / `REJECTED` | 매장매니저관리자(`STORE_ADMIN`) — M7 |
| `PENDING_APPROVAL_L2` (2차 승인 대기) *(v1.7)* | 관리자 최종 승인 대기 | `ACTIVE` / `REJECTED` | 관리자(`ADMIN`) — S3 |
| `ACTIVE` (활성) | 정상 이용 가능 — **로그인 가능 유일 상태** | - | - |
| `REJECTED` (반려) *(v1.7)* | 어느 승인 단계든 거부 | (재신청 시 `REQUESTED`) | 검토자(매장매니저관리자/관리자) |

> - **`USER`**: `EMAIL_VERIFIED → ACTIVE` 로 **직행**한다(승인 분기 없음).
> - **`MANAGER`·`STORE_ADMIN`**: `EMAIL_VERIFIED → PENDING_APPROVAL_L1 → PENDING_APPROVAL_L2 → ACTIVE` 의 **2단계 승인**을 거친다.
> - 코드의 `ApprovalStatus`(`SUBMITTED→APPROVED_L1→CONFIRMED→REJECTED`)와 대응: `SUBMITTED`≈`PENDING_APPROVAL_L1` 진입, `APPROVED_L1`≈`PENDING_APPROVAL_L2` 진입, `CONFIRMED`≈`ACTIVE`.

## 5. 매장·베이·슬롯 도메인 모델

### 5.1 매장 (Store)
- 한 매장은 **N개의 베이**(A1 ~ AN)를 보유한다.
- 운영 시간: **기본 24시간 / 365일**. 단, **매장 휴일 설정** 및 **매니저 휴무**가 존재한다.
  - 매장 운영 시간(24시간)은 변경하지 않는다. **매니저 근무는 3교대로 운영**되며(5.5 참조), 매니저 휴무는 전일 또는 교대조 단위로 발생할 수 있다.

### 5.2 슬롯 (Slot) 정의
> **슬롯 = (매장, 베이, 날짜, 30분 시간단위)** 의 조합. 이 조합은 시스템 전체에서 유일(UNIQUE)하다.

- 시간 단위: **30분** 고정.
- 동일 시간대 최대 수용 인원 = **해당 매장의 베이 수(N)**.
  - 예: 베이가 A1~A3(3개)인 매장은 09:00~09:30 시간대에 최대 3명 예약 가능.

> **베이 슬롯 점유 시 선택 불가 (UX 규칙)**: 슬롯 UNIQUE 제약을 사용자 화면 규칙으로 명문화한다.
> 특정 베이가 특정 `(날짜, 시간)` 슬롯에서 이미 **예약(RESERVED)** 또는 **세차완료(COMPLETED)** 상태이면,
> 그 슬롯에서 해당 베이는 **선택 불가(비활성)** 로 표시되어야 한다. 슬롯 status 전이 및 동시성 처리는 7장과 연계한다.

### 5.3 엔티티 관계 개략도
```
사용자(User) ──< 예약(Reservation) >── 슬롯(Slot)
                      │                     │
                      │                  베이(Bay) ──< 매장(Store)
                      │                                  │
매니저(Manager) ──────┤                          매장 휴일(StoreHoliday)
   │                  │
매니저 휴무(ManagerDayoff)                       가격(Price: 차종 × 서비스)
                      │
                      └── 후기/평점(Review) >── 사용자 / 매장 / 매니저
```

### 5.4 주요 엔티티 요약
| 엔티티 | 핵심 속성(개략) | 관계 |
|--------|----------------|------|
| User | id, email, role, 가입상태 | Reservation 1:N |
| Store | id, name, bayCount | Bay 1:N, Manager 1:N |
| Bay | id, storeId, code(A1~AN) | Slot 1:N |
| Manager | id, storeId, **role(`MANAGER`/`STORE_ADMIN`)**, 가입승인상태(`ApprovalStatus`) | Reservation(대행) 1:N. *(v1.7)* 매장매니저관리자(`STORE_ADMIN`)는 일반매장매니저 기능 전부 + 가입 1차 승인 + 휴가/반차 승인 권한. ~~*(v1.6: `isStoreAdmin` 속성 제거)*~~ *(deprecated — v1.6)* |
| Slot | storeId, bayId, date, timeSlot (UNIQUE) | Reservation 1:1 |
| Reservation | id, userId, slotId, status, 결제방식 | - |
| Review | id, userId, storeId, managerId, rating(1~5), text | 예약완료 사용자만 작성 |
| Price | carType, serviceType, amount | 차종 × 서비스 매트릭스 |
| StoreHoliday | storeId, date | 도메인 데이터(등록 주체 추후 확정/관리자 관리). 슬롯 가용성에만 반영(§6.1). *(v1.6: 승인상태 제거)* |
| ManagerDayoff | managerId, date, **휴무유형(FULL_DAY / SHIFT_1·2·3)**, **승인상태(1단계)** | *(v1.7)* 휴가/반차 신청 → 매장매니저관리자(`STORE_ADMIN`) **1단계 승인으로 종결**(관리자 개입 없음, §5.5·§8.3 참조). 휴무유형은 전일 또는 교대조 단위(5.5 참조). 승인된 휴무만 슬롯 가용성(§6.1)에 반영. ~~*(v1.6: 승인상태 제거, 도메인 데이터로만 유지)*~~ *(deprecated — v1.6)* |

### 5.5 매니저 근무 교대(3교대) & 휴무 유형 *(v1.3 추가)*

매장 운영은 **24시간 유지**(5.1 변경 없음)하되, 매니저 근무는 **3교대** 기준으로 운영한다.

#### 5.5.1 교대 정의
| 교대 | 코드 | 시간대 |
|------|------|--------|
| 1교대(오전조) | `SHIFT_1` | 06:00 ~ 14:00 |
| 2교대(오후조) | `SHIFT_2` | 14:00 ~ 22:00 |
| 3교대(야간조) | `SHIFT_3` | 22:00 ~ 06:00(익일) |

#### 5.5.2 휴무 유형
> ※ 초안에서 처음 언급된 '오전/오후 반차' 개념을 **3교대 기반 부분 휴무로 일반화**한 것이다.

| 휴무 유형 | 코드 | 의미 | 예약 영향 |
|-----------|------|------|-----------|
| 전일 휴무 | `FULL_DAY` | 해당 날짜 종일 휴무 | 그날 **전체 슬롯 비활성** |
| 교대조 단위 휴무 | `SHIFT_1` / `SHIFT_2` / `SHIFT_3` (1개 이상) | 특정 교대 시간대만 휴무 | 해당 교대 시간대의 **30분 슬롯만 비활성**, 그 외 시간대는 예약 가능 |

- 한 날짜에 교대조 휴무를 **복수 지정**할 수 있다(예: `SHIFT_1` + `SHIFT_2`).
- 휴무 유형은 ManagerDayoff 엔티티의 속성으로 보관한다(5.4 참조).

#### 5.5.3 휴가/반차 신청·승인 흐름 *(v1.7)*
- **신청 주체**: 일반매장매니저(`MANAGER`)가 휴가/반차(전일 `FULL_DAY` 또는 교대조 부분 휴무)를 신청한다. 매장매니저관리자(`STORE_ADMIN`)도 본인 휴가/반차를 신청할 수 있다.
- **승인 주체·단계**: **매장매니저관리자(`STORE_ADMIN`)의 1단계 승인/반려로 종결**한다. **관리자(`ADMIN`)는 개입하지 않는다.**
- ※ **단계 수 주의**: 가입 승인은 **2단계**(매장매니저관리자 1차 → 관리자 2차, §4.4)인 반면, 휴가/반차 승인은 **1단계**(매장매니저관리자 종결)다. 이 차이를 문서 전반에서 일관되게 유지한다(§8.3 상태 머신 참조).
- 승인된 휴무만 슬롯 가용성(§6.1)에 반영되어 해당 슬롯을 비활성화한다.

## 6. 예약 기능 (FO 핵심 — 1차 구현)

### 6.1 사용자 예약
- **가입/승인된 매장만** 예약 가능.
- **매니저 휴무(전일/교대조) 시간대의 슬롯은 예약 불가** (해당 매니저 지정 시 휴무 슬롯 비활성).
  - 전일 휴무(`FULL_DAY`)면 그날 전체 슬롯이 비활성, 교대조 휴무(`SHIFT_1`/`SHIFT_2`/`SHIFT_3`)면 해당 교대 시간대의 30분 슬롯만 비활성된다(5.5 참조).
- **베이 슬롯 점유 시 선택 불가**: 특정 베이가 특정 `(날짜, 시간)` 슬롯에 이미 예약(RESERVED)/세차완료(COMPLETED)이면 그 슬롯에서 해당 베이는 비활성으로 표시한다(5.2 정의 및 7장 동시성 연계).
  - 선택 순서는 **매장→매니저→차종→서비스(1페이지) → 날짜·시간 선택 → 그 시간대 베이 그리드에서 점유 베이 비활성·가용 베이 버튼 선택(2페이지)** 이다 *(v1.4 갱신)*. 베이 점유 충돌은 날짜·시간이 정해진 뒤 판정 가능하므로, **날짜·시간 선택 후 해당 시간대의 점유된 베이를 비활성**으로 표시한다(슬롯 그리드 UX). 단계별 화면 분할은 6.5 참조.

### 6.2 매니저 예약 대행
- 로그인한 매니저의 **소속 매장 정보 기준**으로 사용자 예약을 대행한다.
- 매니저 본인의 **휴무를 반영**하여 예약 가능 슬롯을 제한한다.
- **날짜·시간 입력은 일반사용자 예약(§6.5 2페이지)과 동일한 휠 선택기를 사용한다** *(v1.10)*. 자유 텍스트(`YYYY-MM-DD`/`HH:mm`) 입력이 아니라 **오늘부터 21일 날짜 휠 + 30분 단위 시간 휠**이며, 선택한 대행 매니저의 휴무를 동일하게 비활성 처리한다(전일 휴무=날짜 휠 비활성, 교대조 휴무=시간 휠 비활성, §5.5·§6.1 연계).

### 6.3 검색 UX 요건
- 사용자 검색 / 매니저 검색은 **select 박스**로 제공한다.
- select 박스에 **text 입력 시 필터 검색**(자동 필터링)이 동작해야 한다.

### 6.4 결제
- **현장결제만** 지원 (예약 시점에는 결제 미수행, 매장 방문 시 결제).
- 카드결제는 추후 과제 (Out-of-Scope, 2.3 참조).

### 6.5 예약 화면 구성 (3단계 위저드) *(v1.4 추가)*

> 이해관계자 결정으로 예약 플로우를 **3개 화면(라우트)으로 분할한 위저드**로 구성한다.
> 단계 간 진행 데이터는 **클라이언트 상태(Pinia 스토어)** 에 보관하며, 1단계 구현에서는
> **in-memory Pinia만** 사용한다(데이터 진화 1단계, 12.3 연계).

#### 6.5.1 단계별 화면 정의
| 단계 | 라우트 | 화면 역할 | 선택/입력 항목 | 다음 액션 |
|------|--------|-----------|----------------|-----------|
| 1페이지 | `/reserve` | 예약 기본 정보 선택 | 매장 → 매니저 → 차종 → 서비스 선택 (서비스 선택 시 **가격 자동 표시**, 10장 매트릭스) | 4개 선택 완료 시 **"다음"** |
| 2페이지 | `/reserve/slot` | 슬롯·베이 선택 | 날짜·시간 선택 → **해당 시간대 베이 점유 그리드**에서 가용 베이 버튼 선택 | **"예약하기"** 로 확정 (충돌 시 재선택 토스트) |
| 3페이지 | `/reserve/done` | 예약 완료 | 예약 완료 요약 표시 | - |

#### 6.5.2 베이 선택 방식 (단일화)
- 베이 선택은 **버튼 그리드 단일 방식**으로 한다.
- 기존 검색 select box(SearchableSelect) 방식의 베이 선택은 **폐지**한다(중복 선택 제거).
- 베이 점유(RESERVED/COMPLETED) 비활성 규칙(5.2·6.1)은 그대로 유지하며, 점유 베이는
  그리드에서 비활성, 가용 베이만 선택 가능한 버튼으로 노출한다.

> ※ 매장·매니저 검색 UX(6.3, select 박스 + text 필터)는 그대로 유지된다.
> 단일화 대상은 **베이 선택**에 한한다.

#### 6.5.3 진행 데이터 영속성 & 진입 가드
- 1단계 구현은 **in-memory Pinia만** 사용한다 → **브라우저 새로고침 시 진행 데이터는 초기화**된다.
- 미완료 단계로 직접 진입(예: 1페이지를 거치지 않고 `/reserve/slot` 진입)하는 경우
  **1페이지(`/reserve`)로 되돌리는 가드**를 둔다.
- 추후 서버/Redis/JWT 기반 진행상태 영속화는 데이터 진화 2·3단계 과제로 이관한다(12.3 연계).

### 6.6 예약 목록 역할별 분기 *(v1.10 신설 / v1.11 구현)*

> **배경**: 매니저의 예약 목록을 "매니저 본인이 고객으로서 예약한 건(userId)"으로 오해해 노출하던 것을
> **"그 매니저가 담당하는 예약(managerId)"** 으로 바로잡고, 예약 목록 화면(`/reservations`)을 **역할별로 분기**한다.

- 예약 목록 화면 `/reservations`는 로그인 **역할(role)에 따라 노출 범위가 분기**된다(동일 URL, 콘텐츠 분기).
  - **일반사용자(USER)**: 본인이 예약한 건(`userId` 기준) — 기존 동작 유지(§6.1·§11.3).
  - **일반매니저(MANAGER)**: **내 예약**(본인 `userId`) + **담당 예약**(`managerId` 기준)을 **2탭**으로 분리 노출.
    담당 예약에는 다음 **두 경로**가 모두 포함된다.
    1. **매니저가 직접 대행 등록**한 예약(§6.2) — 대행 시 지정한 대행자 매니저 기준.
    2. **일반사용자가 예약 시 해당 매니저를 지정**한 예약(§6.1) — 1페이지 매니저 선택 기준.
  - **매장매니저관리자(STORE_ADMIN)**: 소속 **매장 전체 예약**(`storeId` 기준 — 매장 내 모든 매니저 예약 건)을 노출.
  - **관리자(ADMIN)**: **담당 예약 단일 뷰** — 소속 매장이 없으므로 **매장을 선택**하면 그 매장의 예약만 노출한다.
    기존 매장별 예약 조회(`GET /api/admin/stores/{id}/reservations`, S4)를 재사용한다. 관리자에게는 본인 예약/내 예약 뷰를 제공하지 않는다.
- 담당/매장 전체 목록은 **읽기 전용**(예약자=고객 노출)이며, 본인 예약 목록만 세차완료(M4)·취소(M5) 액션을 제공한다(§11.3).
- **관리자 메뉴 정리(v1.11)**: 관리자(ADMIN) 네비게이션에서 **"예약"(부킹, `/reserve`)과 "휴일 결재"(`/admin/approvals`) 메뉴를 제거**한다(관리자는 예약을 생성하지 않으며, 예약 목록은 매장 선택형 담당 예약으로 일원화). "예약 목록"·"가입 최종 승인"·"매출"은 유지.
- **데이터 모델 연결(해소됨)**: 로그인 매니저 계정(`users`)과 매니저 엔티티(`manager`)를 잇기 위해
  **`users.manager_id` FK 컬럼을 신설**했다(시드: `manager1`→`mgr1`). 로그인 매니저는 이 컬럼으로 자신의
  `managerId`를 해석해 담당 예약을 조회한다(이름·매장 일치 추정 불가 문제 해소).
- **API**(§12.4): `GET /api/reservations`(본인) / `GET /api/manager/reservations`(담당, `managerId` 해석) /
  `GET /api/store-admin/reservations`(매장 전체, `storeId` 해석). BE에 `ReservationMapper.findByManager` 추가.
- ⚠ **신규 매니저 가입분**: 가입 직후 매니저는 `manager_id`가 NULL이라 담당 목록이 빈 상태다. 승인 후
  매니저 엔티티 연결(운영 절차/관리 화면)은 후속 과제로 남긴다.

## 7. 동시 예약(동시성) 처리 — 권장안

> **배경**: 초안에서 개발자가 직접 "예약이 중복될 경우 데이터 동기화 문제 해결방법?"을 질문했다.
> 데이터 진화 3단계(12장)에 맞춰 단계별 권장안을 제시한다. 최종 목표는 3단계의 DB 유니크 제약이며,
> 1·2단계는 그에 도달하기 위한 점진적 시뮬레이션이다.

### 7.1 단계별 권장안
| 단계 | 데이터 계층 | 권장 기법 | 핵심 동작 |
|------|-------------|-----------|-----------|
| **1단계** | 프론트 더미 | 클라이언트 슬롯 잠금 시뮬레이션 + 낙관적 갱신 | 슬롯 선택 시 즉시 `status='예약중'` 표시. 확정 직전 클라이언트 상태로 충돌 감지 후 재선택 유도 |
| **2단계** | Spring Boot 인메모리 더미 | 슬롯 UNIQUE 제약(매장+베이+날짜+시간) + 락 | **MyBatis** 기반 낙관적 락(version 컬럼 비교 UPDATE) 또는 비관적 락(`SELECT ... FOR UPDATE`) 적용. 동시 요청 시 한 건만 성공 |
| **3단계** | MySQL | 트랜잭션 + 유니크 인덱스 | DB 레벨에서 중복 INSERT를 원천 차단. 충돌 시 사용자에게 재선택 유도 |

### 7.2 단계별 장단점 (1줄 비교)
| 단계 | 장점 | 단점 |
|------|------|------|
| 1단계 | 백엔드 없이 UX 검증 가능 | 클라이언트 신뢰 불가 — 진짜 동시성 보장 못 함 |
| 2단계 | 서버 검증 도입, 실제 락 동작 검증 | 인메모리라 재시작 시 휘발, 분산환경 미고려 |
| 3단계 | DB 유니크 인덱스로 무결성 원천 보장 | 충돌 처리(재선택 유도) UX 필수 구현 |

### 7.3 2단계 락 전략 비교
| 기법 | 적합 상황 | 장점 | 단점 |
|------|-----------|------|------|
| 낙관적 락 (version 컬럼) | 충돌 빈도 낮음 | 락 대기 없음, 처리량 높음 | 충돌 시 재시도 로직 필요. **MyBatis는 `UPDATE ... SET version=version+1 WHERE id=? AND version=?` 의 영향 행 수(0=충돌)로 판정** |
| 비관적 락 (SELECT ... FOR UPDATE) | 충돌 빈도 높음(인기 슬롯) | 충돌 자체를 차단 | 락 대기/데드락 위험, 처리량 저하. **MyBatis 매퍼에 `FOR UPDATE` SQL 직접 작성** |

> **권장**: 슬롯 `UNIQUE(storeId, bayId, date, timeSlot)` 제약을 최종 방어선으로 두고,
> 일반 슬롯은 낙관적 락, 경합이 잦은 슬롯은 비관적 락을 선택적으로 적용한다.

> **UX 연계(6.1)**: 슬롯이 `RESERVED`/`COMPLETED` 상태로 전이되면, 해당 `(날짜, 시간)`에서 그 베이는
> 선택 불가(비활성)로 노출되어야 한다(6.1 "베이 슬롯 점유 시 선택 불가" 규칙). 위 동시성 처리는 확정 시점의
> 충돌을 방지하고, 본 UX는 선택 단계에서 점유 베이를 사전 차단하여 충돌 빈도 자체를 낮춘다.

```
FW2(로그인) → FW3(매장선택) → FW5(매장예약) → FW7(예약취소)
```
> 초안 `FW2 → FW3 → M6 → FW6`을 정규화. *(v1.6: 매장예약 승인 M6 제거)* 취소=FW7.

**(c) 예약취소 케이스 2 (승인 전 취소)**
```
FW2(로그인) → FW3(매장선택) → FW7(예약취소)
```
> 초안 `FW2 → FW3 → FW6`을 정규화 (취소=FW7).

**(d) 기존 사용자**
- 최초 사용자 플로우에서 **FW1(회원가입)을 생략**하고 진행.

**(e) 매니저 가입 2단계 승인** *(v1.7)*
```
M1(매니저 가입신청) → [이메일 인증] → M7(매장매니저관리자 1차 승인) → S3(관리자 2차 최종 승인) → ACTIVE
```
> 1차(M7) 또는 2차(S3) 단계에서 반려 시 `REJECTED`로 전이된다(§4.4). `ACTIVE` 이전에는 로그인 불가.

**(f) 휴가/반차 1단계 승인** *(v1.7)*
```
M6(일반매장매니저 휴가/반차 신청) → M8(매장매니저관리자 승인) → 휴무 확정·슬롯 반영(§6.1)
```
> 가입 승인(e, 2단계)과 달리 휴가/반차는 **매장매니저관리자 1단계로 종결**(관리자 개입 없음, §8.3). 반려 시 재신청 가능.
