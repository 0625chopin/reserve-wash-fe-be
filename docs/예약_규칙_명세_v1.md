본 문서는 개발자(사용자)가 구두로 전달한 **2가지 원시 규칙**을 정식 기획 문서로 재편성한 것이다.
원시 규칙(개발자 노트)을 충실히 보존하며, 정본·현재 구현과의 충돌·모호점은 추측하지 않고
**8장 미해결 질문**으로 명시한다.

| # | 원시 규칙(개발자 노트 원문) | 본 문서 매핑 |
|---|------|------|
| 규칙 1 | "매장 → 매니저 → 차종·베이 → 날짜/시간 순으로 선택할 경우에만 예약이 가능하다" | 4.1 기능 요구사항 / 5.1 순차 선택 상태 흐름 |
| 규칙 2 | "베이는 차종(차 크기)에 따라 노출이 다르다. 소/중/대형 등. 소형은 A1, 중형은 B1, 대형은 C1, 특대형은 D1" | 4.1 기능 요구사항 / 5.2 차종↔베이 매핑 표 / 8장 미해결 질문 |

> 미해결 질문으로 분류한다.

## 1. 개요 (Overview)

### 1.1 배경
자동차 세차 예약 서비스(MVP, Nuxt 4 프론트엔드)의 예약 플로우(정본 6장, ROADMAP Phase 4·5)는
이미 **매장 → 매니저 → 차종·베이 → 날짜/시간** 순서로 입력을 받도록 구현되어 있다.
이번에 개발자는 이 플로우에 대해 **두 가지 규칙을 명문화**하고자 한다.

1. **예약 순서 규칙(순차 게이팅)**: 위 순서대로 선택했을 때**만** 예약이 가능하도록 단계 간
   선행조건을 강제한다.
2. **베이 노출 규칙(차종↔베이 매핑)**: 베이가 차종(차 크기)에 따라 다르게 노출되며,
   베이 코드 자체가 크기를 의미하도록 한다(소형→A1, 중형→B1, 대형→C1, 특대형→D1).

### 1.2 목적
- 예약 입력 순서를 **명시적 규칙**으로 정의하여, 잘못된 순서/불완전한 선택으로 인한 예약을 차단한다.
- 차종과 베이의 관계를 **표준 매핑 표**로 정의하여, 사용자가 차 크기에 맞는 베이만 보도록 한다.
- 규칙 2가 **현재 구현과 충돌**하는 지점을 명확히 드러내, 확정 전 의사결정을 돕는다.

### 1.3 해결하려는 문제
| 문제 | 규칙 |
|------|------|
| 사용자가 단계를 건너뛰거나 일부만 선택한 채 예약을 시도할 수 있다 | 규칙 1(순차 게이팅) |
| 차 크기에 맞지 않는 베이가 노출되어 잘못된 예약이 발생할 수 있다 | 규칙 2(차종↔베이 매핑) |
| 베이 식별 체계(현행 `A1/A2/A3` 일련번호)와 크기 의미(`A/B/C/D`)가 불일치 | 규칙 2 + 8장 미해결 질문 |

## 5. 상세 명세 (Detailed Specification)

### 5.1 규칙 1 — 순차 선택 상태 흐름

예약 입력을 **5단계 상태**로 모델링한다. 각 단계는 **선행조건(precondition)** 을 만족해야 진입(노출/활성)한다.
이는 현재 `app/pages/reserve.vue`의 `v-if` 노출 조건 및 `canPickSlot`/`canReserve` computed와 정합한다.

| 단계 | 상태 | 노출/활성 선행조건 | 산출 선택값 | 다음 단계 |
|------|------|---------------------|--------------|-----------|
| ① 매장 선택 | `SELECT_STORE` | (없음 — 진입점) | `storeId` (승인된 매장만, 정본 6.1) | ② |
| ② 매니저 선택 | `SELECT_MANAGER` | `storeId` 선택됨 | `managerId` (해당 매장 매니저만, 정본 6.3) | ③ |
| ③ 차종 선택 | `SELECT_CARTYPE` | `storeId` 선택됨 | `carType` (정본 10.1) | ④ |
| ④ 베이 선택 | `SELECT_BAY` | `storeId` **및** `carType` 선택됨 | `bayId` (차 크기에 맞는 베이만 — 규칙 2) | ⑤ |
| ⑤ 날짜·시간 선택 | `SELECT_SLOT` | `managerId` **및** `bayId` 선택됨(`canPickSlot`) | `date`, `timeSlot` (매니저 휴무일 비활성, 정본 6.1) | 확정 |
| — | `READY_TO_RESERVE` | 6개 값 전부 선택됨(`canReserve`) | — | "예약하기" 활성 |

#### 상태 흐름도

```
[SELECT_STORE]
     │ storeId 선택
     ▼
[SELECT_MANAGER] ──┐
     │ managerId   │ (매장 선택 시 매니저·차종 진입 가능)
     │             │
[SELECT_CARTYPE] ◄─┘
     │ carType 선택
     ▼
[SELECT_BAY]  ← 차 크기에 맞는 베이만 노출 (규칙 2)
     │ bayId 선택  (선행: managerId + bayId 모두 충족 시 다음)
     ▼
[SELECT_SLOT] ← 날짜·시간 (매니저 휴무일 비활성)
     │ date + timeSlot 선택
     ▼
[READY_TO_RESERVE] → "예약하기" 활성 → 예약 확정
```

#### 선행 단계 변경 시 초기화 규칙 (NFR-1, 현행 구현 준수)

| 변경된 단계 | 초기화 대상 |
|-------------|-------------|
| 매장 변경 | 매니저, 베이 (차종은 매장 무관이라 유지) |
| 차종 변경 | 베이 (차 크기에 따라 베이 목록이 달라지므로) |
| 매니저 변경 | 날짜, 시간 (휴무 변동) |
| 베이 변경 | (확정 요약만 초기화) |

> **현행 구현과의 미세 차이(미해결 질문 후보, 8장 Q6)**: 현재 `reserve.vue`는 매장 선택 시
> **매니저와 차종을 동시에 노출**(`v-if="selectedStoreId"`)하고, 베이만 차종 선택을 선행조건으로 둔다.
> 규칙 1의 문언("매니저 → 차종·베이")이 **매니저를 차종보다 반드시 먼저** 선택해야 하는 엄격한 직렬
> 게이팅을 의미하는지, 아니면 "매장 다음에 매니저·차종을 자유 순서로, 단 베이는 차종 이후"인지는 확인이 필요하다.

### 5.2 규칙 2 — 차종 ↔ 베이 매핑 표

#### 5.2.1 사용자 제공 사실 (원문 보존 — 확정)

| 차 크기 등급 | 베이 코드 |
|--------------|-----------|
| 소형 | A1 |
| 중형 | B1 |
| 대형 | C1 |
| 특대형 | D1 |

> 위 표는 **개발자 노트 원문 그대로**이다. 베이 코드의 알파벳(A/B/C/D)이 **크기 등급**을 의미하며,
> 기존 정본의 "매장별 일련번호(A1~AN)" 의미와는 **다른 체계**임에 유의한다(8장 Q4).

#### 5.2.2 정본 차종 5분류와의 대조 (확정 매핑 아님 — 미해결)

정본 10.1의 차종은 **5분류**, 규칙 2의 베이 크기는 **4등급**이다. 등급 수가 달라 1:1 대응이 성립하지 않으며,
아래 표의 "베이 매핑" 열은 **확정값이 아니라 미해결 질문(8장 Q1)** 임을 명시한다.

| 정본 차종 코드 | 분류(정본 10.1) | 현행 `Bay.size`(3등급) | 규칙 2 베이 등급(4등급) | 베이 코드 |
|----------------|------------------|------------------------|-------------------------|-----------|
| `LIGHT` | 경형 | SMALL | **? (소형?)** | **? (A1?)** |
| `SMALL` | 소형 | SMALL | **? (소형?)** | **? (A1?)** |
| `MID` | 준중형·중형 | MID | **? (중형?)** | **? (B1?)** |
| `LARGE` | 대형·SUV | LARGE | **? (대형?)** | **? (C1?)** |
| `VAN_ETC` | 승합·기타 | LARGE | **? (특대형?)** | **? (D1?)** |

> 위 `?` 표시는 **추측 후보**일 뿐 확정이 아니다. 정본은 차 크기를 3등급(SMALL/MID/LARGE)으로,
> 규칙 2는 4등급(소/중/대/특대)으로 다루므로, **특대형(D1)이 신설**되며 5개 차종을 4등급에 배분하는
> 방식이 정의되어야 한다(8장 Q1, Q5).

#### 5.2.3 현재 구현 동작 (충돌 분석용 — 변경 전 기준)

현재 코드(`app/services/storeService.ts`, `app/data/stores.ts`)의 동작은 규칙 2와 다음과 같이 충돌한다.

| 항목 | 현재 구현 | 규칙 2(신규) |
|------|-----------|--------------|
| 베이 코드 의미 | 매장별 **일련번호** `A1/A2/A3` | 크기 의미 `A=소/B=중/C=대/D=특대`, `A1/B1/C1/D1` |
| 베이 크기 등급 | **3등급** `BaySize = SMALL\|MID\|LARGE` | **4등급** 소/중/대/특대 (특대 신설) |
| 차→베이 노출 방식 | **누적 수용**: 차가 요구하는 크기 **이상**을 수용하는 모든 베이(`SIZE_RANK[b.size] >= min`). 소형차→전체 베이, 대형차→LARGE 베이만 | **차 크기별 특정 베이**처럼 보임(소형→A1 등). **1:1 매칭인지 누적인지 불명**(8장 Q3) |
| 매장별 베이 구성 | 매장마다 상이(강남점 A1~A3, 홍대점 A1~A2 등) | 크기 코드 체계로 통일 시 매장별 베이 보유 규칙 재정의 필요(8장 Q2, Q4) |

> 즉 규칙 2를 그대로 적용하려면 **베이 식별 체계·크기 등급 수·노출 알고리즘·더미 데이터**를 모두
> 손봐야 하며, 이는 정본 5장(도메인 모델)과 `app/types/enums.ts`의 `BaySize` 타입 변경을 수반한다.
> **단, 본 문서는 기획 문서이므로 코드는 변경하지 않는다.** 변경 범위는 확정(8장) 이후 ROADMAP/구현에서 다룬다.

### 5.3 예외 처리
